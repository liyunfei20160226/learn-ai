# PRD 生成器

创建清晰、可执行、适合开发的详细产品需求文档。

---

## 工作流程

1. 接收用户的功能描述
2. 提出 3-5 个必要的澄清问题（带字母选项）
3. 根据回答生成结构化的 PRD
4. 保存到 `tasks/prd-[功能名称].md`

**重要提示：** 不要开始实现代码。只创建 PRD 即可。

---

## 第一步：澄清问题

只在初始描述模糊时提出关键问题。重点关注：

- **问题/目标：** 这个功能解决什么问题？
- **核心功能：** 关键操作是什么？
- **范围/边界：** 哪些事情不应该做？
- **验收标准：** 我们如何知道完成了？

### 问题格式示例：

```
1. 这个功能的主要目标是什么？
   A. 改善用户入门体验
   B. 提高用户留存率
   C. 减少支持负担
   D. 其他：[请说明]

2. 目标用户是谁？
   A. 仅新用户
   B. 仅现有用户
   C. 所有用户
   D. 仅管理员用户

3. 范围是什么？
   A. 最小可行版本
   B. 全功能实现
   C. 仅后端/API
   D. 仅UI
```

这样用户可以用 "1A, 2C, 3B" 这样的方式快速回答。记住选项要缩进。

---

## 第二步：PRD 结构

生成的 PRD 需要包含以下章节：

### 1. 介绍/概述
简要描述功能和它解决的问题。

### 2. 目标
具体、可衡量的目标（项目符号列表）。

### 3. 用户故事

每个故事需要：
- **标题：** 简短描述性名称
- **描述：** "作为一个[用户]，我想要[功能]，以便[收益]"
- **验收标准：** 可验证的检查清单，明确"完成"的定义

每个故事应该足够小，可以在一个专注的会话中完成。

**格式：**
```markdown
### US-001: [标题]
**描述：** 作为一个[用户]，我想要[功能]，以便[收益]。

**验收标准：**
- [ ] 具体可验证的标准
- [ ] 另一个标准
- [ ] 类型检查/lint 通过
- [ ] **[仅UI故事]** 使用 dev-browser 技能在浏览器中验证
```

**重要提示：**
- 验收标准必须可验证，不能模糊。"正常工作"不好。"删除按钮会显示确认对话框"很好。
- **对于任何涉及 UI 修改的故事：** 始终包含"使用 dev-browser 技能在浏览器中验证"作为验收标准。这确保前端工作得到视觉验证。

### 4. 功能需求
编号的具体功能列表：
- "FR-1：系统必须允许用户..."
- "FR-2：当用户点击 X 时，系统必须..."

表述要明确无误。

### 5. 非目标（超出范围）
这个功能不包含的内容。这对管理范围至关重要。

### 6. 设计考虑（可选）
- UI/UX 需求
- 如果有原型，附上链接
- 可以复用的相关现有组件

### 7. 技术考虑（可选）
- 已知的约束或依赖
- 与现有系统的集成点
- 性能需求

### 8. 成功指标
如何衡量成功？
- "将完成 X 的时间减少 50%"
- "提高转化率 10%"

### 9. 未解决问题
仍需要澄清的问题或领域。

---

## 面向初级开发者的写作规范

PRD 的读者可能是初级开发者或 AI 代理。因此：

- 表述要明确无误
- 避免行话，或解释清楚
- 提供足够的细节来理解目的和核心逻辑
- 给需求编号，方便引用
- 在有帮助的地方使用具体示例

---

## 输出

- **格式：** Markdown (`.md`)
- **位置：** `tasks/`
- **文件名：** `prd-[功能名称].md`（kebab-case 短横线命名法）

---

## PRD 示例

```markdown
# PRD: 任务优先级系统

## 介绍

为任务添加优先级级别，让用户可以专注于最重要的事情。任务可以标记为高、中、低优先级，通过视觉指示器和过滤帮助用户有效管理工作负载。

## 目标

- 允许给任何任务分配优先级（高/中/低）
- 提供不同优先级之间清晰的视觉区分
- 支持按优先级过滤和排序
- 新任务默认中等优先级

## 用户故事

### US-001: 在数据库中添加优先级字段
**描述：** 作为开发者，我需要存储任务优先级，以便在会话间持久化。

**验收标准：**
- [ ] 在 tasks 表添加优先级列: 'high' | 'medium' | 'low'（默认 'medium'）
- [ ] 成功生成并运行迁移
- [ ] 类型检查通过

### US-002: 在任务卡片上显示优先级指示器
**描述：** 作为用户，我希望一眼就能看到任务优先级，以便知道需要先关注什么。

**验收标准：**
- [ ] 每个任务卡片显示彩色优先级徽章（红=高，黄=中，灰=低）
- [ ] 无需悬停或点击就能看到优先级
- [ ] 类型检查通过
- [ ] 使用 dev-browser 技能在浏览器中验证

### US-003: 在任务编辑中添加优先级选择器
**描述：** 作为用户，我希望在编辑任务时更改优先级。

**验收标准：**
- [ ] 任务编辑弹窗中有优先级下拉框
- [ ] 当前优先级显示为选中状态
- [ ] 选择更改后立即保存
- [ ] 类型检查通过
- [ ] 使用 dev-browser 技能在浏览器中验证

### US-004: 按优先级筛选任务
**描述：** 作为用户，我希望在专注时只查看高优先级项目。

**验收标准：**
- [ ] 筛选下拉框选项：全部 | 高 | 中 | 低
- [ ] 筛选状态持久化在 URL 参数中
- [ ] 没有匹配任务时显示空状态消息
- [ ] 类型检查通过
- [ ] 使用 dev-browser 技能在浏览器中验证

## 功能需求

- FR-1：在 tasks 表添加 `priority` 字段 ('high' | 'medium' | 'low', 默认 'medium')
- FR-2：在每个任务卡片显示彩色优先级徽章
- FR-3：在任务编辑弹窗中包含优先级选择器
- FR-4：在任务列表头部添加优先级筛选下拉框
- FR-5：在每个状态列中按优先级排序（高→中→低）

## 非目标

- 不包含基于优先级的通知或提醒
- 不包含基于截止日期的自动优先级分配
- 不包含子任务的优先级继承

## 技术考虑

- 复用现有带颜色变体的徽章组件
- 筛选状态通过 URL 搜索参数管理
- 优先级存储在数据库中，不动态计算

## 成功指标

- 用户可以在 2 次点击内完成优先级更改
- 高优先级任务立即在列表顶部可见
- 任务列表性能没有退化

## 未解决问题

- 优先级是否会影响列内的任务排序？
- 是否需要为优先级更改添加键盘快捷键？
```

## 检查清单

保存 PRD 之前：

- [ ] 已提出带字母选项的澄清问题
- [ ] 已整合用户的回答
- [ ] 用户故事小而具体
- [ ] 功能需求已编号且明确无误
- [ ] 非目标章节定义了清晰的边界
- [ ] 已保存到 `tasks/prd-[功能名称].md`
