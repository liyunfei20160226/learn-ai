## 1. 概述

### 1.1 项目背景
当前业务场景中存在大量的销售票据、邮政受理业务与物流小箱容器管理需求。现有的处理方式依赖人工操作或分散的系统，导致数据录入效率低下、状态追踪不透明，且容易产生人为错误。此外，邮政受理作为独立的核心业务板块，缺乏系统化支持。为了提升业务处理效率，确保票据信息的准确性、业务流程的可追溯性以及物流容器状态的实时可控性，需要开发一套集中的票据管理系统。

### 1.2 问题定义
目前业务面临以下主要痛点：
1.  **录入繁琐与模式缺失**：缺乏统一的票据受理入口，手入力与大量数据导入缺乏高效工具支持；同时缺乏对"通常/INFOX"特定业务模式的支持，导致部分业务场景无法覆盖。
2.  **状态黑盒与资产脱节**：小箱（物流容器）的在库、运输、空箱状态无法实时统一查看；缺乏小箱资产的初始化与废弃管理入口，导致数据来源缺失且无法处理废旧资产；业务处理工序状态缺乏系统记录；缺乏对容器状态的校验与自动流转，易导致资产重复分配或状态不一致。
3.  **数据孤岛与修正困难**：票据信息与实物容器状态未实现有效关联；缺乏对已录入票据的修改与作废机制，难以应对录入错误或业务变更，导致打印清单校验逻辑无法闭环。
4.  **基础配置缺失**：缺乏"终端组"、"制造商"、"规格型号"及"客户"等基础主数据管理，导致系统无法自动填充"端末识别番号"等关键字段，且客户名称录入不一致（如"ABC公司" vs "ABC有限公司"），规格型号录入不统一，影响数据准确性和查询效率。
5.  **业务互斥与冲突风险**：销售业务与邮政业务共享小箱资产，缺乏明确的互斥规则，可能导致同一容器被跨业务重复占用。
6.  **安全与合规风险**：票据数据涉及客户隐私信息，现有流程缺乏脱敏与加密机制，存在数据泄露风险；缺乏明确的数据归档与灾难恢复策略，长期运行将面临性能瓶颈与数据丢失风险。
7.  **跨部门数据管理缺失**：缺乏按部门/组织机构的数据隔离机制，不同部门数据可互相访问，不符合企业分级管理要求。
8.  **历史数据迁移困难**：旧系统历史数据无法平滑迁移，导致业务断档或数据不完整。

### 1.3 项目目标
构建一套标准化的票据管理系统，实现：
1.  支持销售票据及邮政受理业务的多模式录入（手入力、大量导入、INFOX模式），提升录入效率 50% 以上。
2.  实现小箱资产的全生命周期管理（从初始化入录、状态自动流转到废弃处理）、业务工序状态的自动流转及资产状态校验，支持同一容器下的业务追加录入，防止跨业务类型冲突，确保账实相符。
3.  提供标准化的票据清单打印功能，并通过支持票据修改/作废功能，确保物理存档与交接数据的准确性。
4.  构建包含客户主数据、规格型号在内的基础数据管理模块，自动采集终端识别信息，确保数据录入的准确性与一致性。
5.  建立数据安全、归档与备份机制，对个人隐私信息（PII）进行加密存储与脱敏展示，实施"在线保存2年，历史归档"策略，并依托云服务实现数据灾难恢复，确保系统长期稳定运行并符合数据安全合规要求。
6.  支持按部门/组织机构的数据权限隔离，满足大型企业分级管理需求。
7.  提供历史数据迁移能力，确保旧系统数据平滑过渡，保障业务连续性。

## 2. 用户故事

### US-01: 销售票据手动录入
**As a** 数据录入员,
**I want** 通过标准界面手动输入销售票据信息（包括票据ID、客户信息、金额等）,
**So that** 我可以快速处理单笔或少量的票据业务，确保数据进入系统。

**验收标准：**
- [ ] 系统提供"销售票受付入力画面"，包含必填字段校验。
- [ ] 输入完毕点击"保存"后，数据写入数据库，系统自动更新工序状态（如设置为"02"已受理），并提示保存成功。
- [ ] **[更新]** 保存成功的同时，系统自动触发关联小箱主数据的状态变更，确保资产状态实时同步。每次受理成功后保持小箱为"已分配"状态，支持同一小箱在"已分配"状态下多次追加录入不同票据，满足拼箱业务场景需求。当小箱正式出库运输时，由仓库操作员统一变更状态为"运输中"。
- [ ] 点击"Cancel"按钮时，不保存数据，焦点返回一栏表画面，处理终止。
- [ ] 系统根据终端组配置，自动获取并记录"端末识别番号"与"端末グループ番号"；端末识别番号需拆分为"上桁"、"下桁"及"制造商番号"存储。
- [ ] **[更新]** 系统在保存前校验关联小箱的状态，允许"空箱"、"在库"及"已分配"状态的小箱进行受理（支持追加录入）；若状态为"运输中"、"锁定"或"废弃"，系统阻止录入并提示错误信息。
- [ ] **[新增]** 系统校验小箱是否存在未完成的邮政受理业务，若存在则阻止录入，确保销售与邮政业务互斥。
- [ ] **[新增]** 客户名称/ID字段支持通过下拉选择或模糊搜索匹配客户主数据，确保录入一致性。
- [ ] 采用乐观锁机制，当并发修改同一票据或预订同一小箱时，系统阻止操作并提示"数据已被修改，请刷新后重试"。
- [ ] 数据仅能被同部门用户查看和操作，跨部门访问需要特殊授权。

### US-02: 销售票据批量导入
**As a** 业务主管,
**I want** 通过Excel或CSV文件批量上传销售票据数据,
**So that** 我可以在短时间内处理大量历史数据或批量业务，减少重复劳动。

**验收标准：**
- [ ] 系统支持下载标准模板。
- [ ] 上传文件后，系统进行格式校验，异常数据需生成错误报告供下载。
- [ ] 成功导入的数据需在列表中即时可见，并同步更新工序管理数据。
- [ ] 导入过程采用"全部回滚"策略。若文件中存在任何校验错误，整个批次数据均不提交入库，确保数据一致性，并输出包含具体行号和错误原因的报告。
- [ ] MVP版本导入结果直接在页面展示，不提供消息通知。

### US-03: 小箱状况一栏查看
**As a** 仓库管理员,
**I want** 在"小箱状况一栏"画面中查看所有小箱的当前状态（如：在库、出库、运输中、空箱）,
**So that** 我可以实时掌握容器资产情况，合理安排物流资源。

**验收标准：**
- [ ] 列表展示小箱ID、类型/规格、当前状态、最后更新时间等关键信息。
- [ ] 支持按状态筛选和按ID搜索，小箱编号支持模糊搜索/前缀匹配，同时保留精确匹配选项。
- [ ] 数据刷新延迟不超过 1 分钟。
- [ ] 针对百万级数据量，查询响应时间应控制在秒级。
- [ ] 支持分页展示，默认每页显示20条记录，提供选项（20/50/100）让用户自行调整分页大小，支持页码切换。
- [ ] 仅展示当前用户所属部门的数据，跨部门数据需授权才能查看。
- [ ] 提供手动状态变更入口，允许仓库操作员将小箱状态从"已分配"变更为"运输中"，以及从"运输中"变更回"在库"或"空箱"。

### US-04: 票据与状态查询
**As a** 客服人员,
**I want** 通过票据号查询关联的小箱状态,
**So that** 我可以快速回复客户关于物流进度的咨询。

**验收标准：**
- [ ] 在检索框输入票据号，支持模糊搜索/前缀匹配，同时保留精确匹配选项，可匹配到对应记录。
- [ ] 结果页展示票据基础信息及关联的小箱实时状态。
- [ ] 客户敏感信息（如客户名称）在界面展示时需进行脱敏处理。
- [ ] 支持分页展示查询结果，默认每页显示20条记录，提供选项（20/50/100）让用户自行调整分页大小。
- [ ] 授权用户可将查询结果导出为Excel/CSV格式，导出时自动对敏感信息脱敏。
- [ ] 归档数据仍支持查询入口，仅查询性能要求可适当放宽。

### US-05: 邮政受理业务录入
**As a** 邮政业务操作员,
**I want** 在"邮政受理画面"中录入邮政受理信息（受理年月、小箱编号、封筒SEQ等）,
**So that** 我可以独立于销售票据处理邮政相关业务，确保业务完整性。

**验收标准：**
- [ ] 系统提供独立的"邮政受理画面"，包含受理年月、小箱编号、封筒SEQ等特定字段。
- [ ] 保存时校验关键字段的唯一性与格式（如封筒SEQ格式）。
- [ ] **[新增]** 保存成功后，需同步更新相关联的小箱主数据状态，每次受理成功后保持小箱为"已分配"状态，支持同一小箱在"已分配"状态下多次追加录入。
- [ ] **[新增]** 系统校验小箱是否存在未完成的销售票据业务，若存在则阻止录入，确保销售与邮政业务互斥。
- [ ] 系统界面需支持条码扫描枪输入，输入框应自动获取焦点并保持，以适应连续扫描作业的需求。
- [ ] 保存成功后，工序状态（作业工程区分）应更新为特定值"03"（代表邮政受理），以区别于销售票据受理。
- [ ] 行番号生成逻辑依据"同一封筒SEQ内"进行分组计数，系统自动计算当前最大行号并加1，确保同一封筒下的记录序号唯一且连续。
- [ ] 数据仅能被同部门用户查看和操作。

### US-06: 受理票据清单打印
**As a** 业务主管,
**I want** 打印"受理销售票一览表"用于存档或交接,
**So that** 我可以满足线下归档要求并核对实物交接情况。

**验收标准：**
- [ ] 系统提供"印刷"按钮，支持按条件生成一览表。
- [ ] 打印前执行数据校验（如小箱编号范围有效性、修正日期逻辑）。
- [ ] **[更新]** 输出方式调整为生成标准PDF文件供用户下载或通过浏览器打印功能输出，不涉及服务端直接连接打印机指令。

### US-07: 票据修改与作废
**As a** 业务主管,
**I want** 对已录入的票据进行修改或作废操作,
**So that** 我可以纠正录入错误或处理业务变更，确保系统数据与实际业务一致。

**验收标准：**
- [ ] 授权用户可在详情页或列表页发起修改/作废操作。
- [ ] 修改时系统自动记录"修正年月日"及"修正者"信息。
- [ ] 作废操作为逻辑删除，保留历史记录，并同步更新工序管理数据（标记为无效或取消）。
- [ ] **[新增]** 作废票据后，不自动回滚关联小箱状态。小箱状态由实际物理流转决定，应由仓库管理员根据实际物理情况手动调整，保持数据与实物一致。
- [ ] 打印输出时，已作废票据不纳入正常统计，或标记为作废状态。
- [ ] 采用乐观锁机制，更新时检查版本号；若并发冲突发生，系统阻止操作并提示用户"数据已被修改，请刷新后重试"。
- [ ] **[新增]** MVP版本只需要记录"修正年月日"和"修正者"信息，满足基本追溯要求即可。不需要记录每次修改的具体内容差异。

### US-08: 基础数据管理
**As a** 系统管理员,
**I want** 维护终端组、制造商信息、小箱规格型号及其映射关系,
**So that** 系统可以自动、准确地填充票据录入时的终端识别信息，减少人工错误，并确保规格型号数据一致性。

**验收标准：**
- [ ] 提供界面维护"端末グループ番号"、"端末識別番号"与"メーカー番号"的对应关系。
- [ ] 提供界面维护小箱规格型号字典，支持增删改查，确保数据标准化。
- [ ] "端末識別番号"在数据库中应拆分为"上桁"、"下桁"及"制造商番号"三个独立字段存储，以支持按制造商统计等查询需求。
- [ ] 票据录入时，系统根据当前设备或用户所属组，自动匹配并填充相关字段。
- [ ] 若系统无法自动识别终端信息（如IP未配置），应弹出提示框允许用户从已有"终端组"列表中手动选择，作为降级方案确保业务连续性，并记录日志。

### US-09: 小箱资产全生命周期管理
**As a** 资产管理员,
**I want** 录入、导入新小箱或对废弃小箱进行删除/作废操作,
**So that** 业务流转有据可依，且系统中不存在无效的"幽灵容器"数据，保持主数据质量。

**验收标准：**
- [ ] 提供小箱主数据录入界面（小箱ID、规格、所属部门等）。
- [ ] 规格字段通过下拉选择关联基础数据中的规格型号字典，不允许自由录入。
- [ ] 支持批量导入小箱资产。
- [ ] 新增小箱默认状态为"空箱/在库"。
- [ ] 支持对小箱主数据进行逻辑删除（作废）或物理删除。
- [ ] 仅允许删除无业务关联（无历史受理记录）的测试数据或错误数据；对于有历史记录的废弃容器，仅允许作废（状态标记为"废弃"）。
- [ ] 支持"一括情報削除"（批量信息删除）功能，允许管理员批量清理无效的主数据记录。
- [ ] 系统完整支持以下6种小箱状态：空箱、在库、已分配、运输中、锁定、废弃。
- [ ] 支持同一小箱在"已分配"状态下多次追加录入不同票据，满足拼箱业务需求。
- [ ] **[新增]** 当小箱正式出库运输时，由仓库操作员统一在小箱管理界面执行状态变更操作，将状态从"已分配"变更为"运输中"。
- [ ] **[新增]** 小箱完成派送回库后，由仓库管理员手动将状态从"运输中"变更回"在库"或"空箱"，系统记录每次状态变更的操作人、时间。
- [ ] **[更新]** 批量删除（一括情報削除）时，系统自动跳过已关联业务记录的小箱，仅删除无业务关联的小箱；操作完成后输出报告，说明成功删除条数、跳过条数及原因，不强制整个批量操作回滚。
- [ ] **[新增]** 小箱被标记为"废弃"状态后，MVP版本不支持自动化恢复操作；若确需恢复，由管理员手动修改数据库状态，后续迭代可按需添加自动化恢复功能。
- [ ] 小箱数据按部门归属隔离，仅归属部门用户可操作。

### US-10: INFOX模式录入
**As a** 数据录入员,
**I want** 使用"通常/INFOX"模式进行票据录入,

## 3. 功能需求

### 3.1 销售票据受理模块

#### 3.1.1 手入力功能
- **界面设计**：参考《【CCS】画面定義書_売上票受付入力画面 (手入力・大量)_1.0.xlsx》。
- **字段定义**：
  - 票据编号（系统生成或手动输入，唯一键）。
  - **客户名称/ID**：关联客户主数据，支持模糊搜索匹配。
  - 受理日期（默认当前日期，可修改）。
  - 金额/明细。
  - **端末识别番号**：系统根据基础数据配置自动获取，在数据库中需拆分为"上桁"、"下桁"、"制造商番号"存储。
  - **端末グループ番号**（系统根据基础数据配置自动获取，必填）。
  - **所属部门**：自动关联当前用户所属部门，用于数据权限隔离。
- **交互逻辑**：
  - [Save] 按钮：校验通过后保存，更新工程管理数据，提示成功，清空表单。
  - [Cancel] 按钮：不保存，焦点返回主列表，终止处理。
- **小箱状态流转与校验**：
  - **[更新]** 状态校验规则调整：允许"空箱"、"在库"、"已分配"状态的小箱进行受理，以支持同一容器下多笔业务的追加录入。
  - 阻止"运输中"、"锁定"、"废弃"状态的小箱进行受理。
  - **[新增]** 业务互斥校验：受理前检查小箱是否存在未完成的邮政受理业务，若存在则拦截，确保销售与邮政业务资产不冲突。
  - **[更新]** 状态流转规则：每次受理成功后保持小箱为"已分配"状态，直到发生邮政业务或实体出库运输后才变更为"运输中"。支持同一小箱在"已分配"状态下多次追加录入不同票据，满足拼箱业务场景需求。当小箱正式出库运输时，由仓库操作员统一变更状态为"运输中"。小箱完成派送回库后，由仓库管理员手动变更回"在库"或"空箱"。
  - 保存成功后，系统自动触发小箱主数据状态的变更，实现票据与资产状态的联动。系统记录每次状态变更的操作人、时间和变更原因。
- **并发控制**：
  - 采用乐观锁机制，数据库表增加"版本号"字段。
  - 更新或受理时检查版本号，若检测到冲突，阻止操作并提示"数据已被修改，请刷新后重试"。
- **数据权限**：用户只能操作所属部门的数据，跨部门操作需要额外授权。

#### 3.1.2 大量数据导入功能
- 支持文件格式：.xlsx, .csv。
- 数据校验：必填项检查、格式检查、唯一性检查。
- 异常处理：提供"错误行号+错误原因"的反馈文档下载。
- **事务处理策略**：
  - 采用"全部回滚"策略。票据业务具有严肃性，系统应在导入前进行预校验。
  - 一旦发现任何错误，整个批次不予提交，确保数据库状态一致性。
- **并发限制**：限制同时只能进行一个导入任务，防止服务器资源耗尽。
- **结果反馈**：导入结果直接在页面展示成功/失败信息，并提供错误报告下载。MVP版本不发送消息通知。

#### 3.1.3 通常/INFOX模式录入
- **界面设计**：参考《【CCS】画面定義書_売上票受付入力画面（通常・INFOX）_1.0.xlsx》。
- **业务定义**：INFOX定义为系统内部的一种特定录入界面模式。
- **数据处理**：保存逻辑与手入力类似，需额外标记数据来源为"INFOX"。
- **封筒SEQ生成逻辑**：
  - 系统需支持动态切换生成模式。
  - 界面提供选项控件：
    - **选项 0:连番**：系统自动生成流水号。
    - **选项 1:端末识别番号**：根据"端末番号（从/到）"字段范围自动填充或校验封筒SEQ。

#### 3.1.4 票据修改与作废
- **修改功能**：
  - 允许修改未锁定或未完成最终工序的票据。
  - 修改保存时，系统自动更新"修正年月日"、"修正者"字段。
  - 触发工序状态更新。
- **作废功能**：
  - 支持对错误票据进行逻辑删除（作废）。
  - 作废后数据保留在数据库中，状态标记为"无效"。
  - **[新增]** 作废票据后，不自动回滚关联小箱状态。小箱状态由实际物理流转决定，应由仓库管理员根据实际物理情况手动调整，保持数据与实物一致。
- **并发控制**：
  - 修改操作受乐观锁保护。

#### 3.1.5 打印输出功能
- **触发时机**：在受理一览画面点击"印刷"按钮。
- **输出内容**：受理销售票一览表。
- **业务校验**：校验小箱编号范围与修正日期逻辑。
- **[更新] 实现方式**：MVP版本采用生成PDF文件供用户下载或浏览器打印的方式。此方案兼容性好，无需处理复杂的客户端打印机驱动问题，降低技术实现风险。

#### 3.1.6 数据导出功能
- **触发时机**：在列表查询页面点击"导出"按钮。
- **输出格式**：支持Excel(.xlsx)和CSV格式。
- **权限控制**：仅限管理员和业务主管角色可使用。
- **数据安全**：导出时自动对敏感客户信息进行脱敏处理。

### 3.2 邮政受理模块

#### 3.2.1 邮政受理录入
- **界面设计**：参考《【CCS】画面定義書_郵便受付画面》。
- **核心字段**：受理年月、小箱编号、封筒SEQ、所属部门等。
- **硬件支持**：界面需支持条码扫描枪，输入框自动获取焦点。
- **逻辑处理**：
  - 保存时需校验小箱编号是否存在。
  - **[更新]** 业务互斥校验：受理前检查小箱是否存在未完成的销售票据业务，若存在则拦截。
  - **[更新]** 状态校验：遵循与销售票据一致的规则，允许"已分配"状态下的追加录入。每次受理成功后保持小箱为"已分配"状态，支持同一小箱多次追加录入。
  - 保存成功后，需同步更新小箱主数据状态，确保资产流转记录完整。
  - 工序状态更新为"03"。
- **行番号自动生成**：
  - 依据"同一封筒SEQ内"分组计数，系统自动计算最大行号并加1，确保记录序号唯一且连续。
- **数据权限**：用户只能操作所属部门的数据。

### 3.3 小箱状况管理模块

#### 3.3.1 小箱状况一栏表
- **界面设计**：参考《【CCS】画面定義書_小箱状況一覧_1.0.xlsx》。
- **展示列**：小箱编号、规格型号、当前状态、所在位置、关联票据号、更新时间、所属部门。
- **排序与筛选**：支持按更新时间倒序排列，支持按状态多选筛选。
- **搜索支持**：关键字段（小箱编号）支持模糊搜索/前缀匹配，同时保留精确匹配选项。
- **性能要求**：百万级数据量下查询响应控制在秒级。
- **分页展示**：所有列表页面必须支持分页，默认每页显示20条记录，提供选项（20/50/100）让用户自行调整分页大小，支持页码切换。
- **数据权限**：默认仅展示当前用户所属部门的数据，支持跨部门数据查询（需授权）。
- **导出功能**：授权用户可导出查询结果为Excel/CSV，敏感信息自动脱敏。
- **状态变更**：提供手动操作入口，允许仓库操作员将小箱状态从"已分配"变更为"运输中"，以及从"运输中"变更回"在库"或"空箱"。系统记录每次状态变更的操作人、时间和原因。

#### 3.3.2 小箱资产管理
- **资产初始化**：提供小箱基础信息录入功能，支持单笔录入与批量导入。
- **字段定义**：小箱编号（唯一）、规格型号（关联基础数据字典）、归属部门、购入日期、初始状态。
- **状态枚举**：系统完整定义以下6种小箱状态：**空箱、在库、已分配、运输中、锁定、废弃**。
- **状态设置**：新录入小箱默认状态为"在库（空箱）"。
- **资产生命周期管理**：
  - **作废与删除**：支持对无业务关联的测试数据或错误录入数据进行物理删除；对有历史记录但已废弃的容器进行逻辑删除（作废）。
  - **批量操作**：实现"一括情報削除"功能，支持管理员批量处理无效数据，维护主数据质量。
  - **[更新]** 批量删除处理规则：执行批量删除时，系统自动跳过已关联业务记录的小箱，仅删除无业务关联的小箱；操作完成后输出报告，说明成功删除条数、跳过条数及原因，不强制整个批量操作回滚。
  - **[更新]** 支持拼箱业务：允许同一小箱在"已分配"状态下多次追加录入不同票据，满足拼箱业务场景需求。
  - **[新增]** 状态变更操作：当小箱正式出库运输时，由仓库操作员统一执行状态变更，从"已分配"变更为"运输中"。小箱完成派送回库后，由仓库管理员手动变更回"在库"或"空箱"。
  - **[新增] 废弃恢复规则**：小箱标记为"废弃"后，MVP版本不支持自动化恢复，若确需恢复由管理员手动修改数据库状态。
- **数据权限**：小箱按归属部门隔离，仅归属部门用户可操作。

### 3.4 工序管理模块
- **状态流转**：系统在票据录入、修改、作废保存时，自动更新"工程管理数据"。
- **数据记录**：
  - **作业工程区分**：
    - 销售票据受理：设置为"02"。
    - 邮政受理：设置为"03"。
    - 已作废：设置为"99"。
  - **终了日时**：记录操作完成的时间戳。
  - **操作人**：记录当前操作用户。
- **数据权限**：工序数据跟随票据/小箱按部门隔离。

### 3.5 基础数据管理模块
- **终端组管理**：维护"端末グループ番号"及其属性。
- **制造商管理**：维护"メーカー番号"及制造商基础信息。
- **规格型号管理**：维护小箱规格型号标准化字典，确保数据一致性，支持增删改查。
- **[新增] 客户主数据管理**：
  - 提供客户信息的增删改查功能。
  - 作为票据录入时客户字段的唯一数据源，确保数据一致性。
  - **[新增] 删除规则**：当客户已有业务票据关联时，禁止物理删除，仅允许逻辑删除（标记为无效）；对于无业务关联的客户可删除。保留历史关联数据，保证数据追溯性。
- **映射关系配置**：配置终端组与制造商、终端识别番号的对应关系。
- **端末识别番号结构**：
  - 数据库设计中，"端末识别番号"需拆分为"上桁"、"下桁"及"制造商番号"三个物理字段存储。
- **终端识别异常处理**：
  - 当系统无法自动识别终端信息时，提供手动选择机制作为降级方案。
- **数据权限**：基础数据可按部门配置访问权限。
- **数据字典配置**：MVP版本核心业务状态采用硬编码方式实现，不支持在线配置。

### 3.6 系统管理模块
- **用户权限**：区分录入员、管理员、访客。基于RBAC（基于角色的访问控制）实现权限管理。
- **数据权限隔离**：支持按部门/组织机构进行数据隔离，用户默认只能查看和操作本部门数据；跨部门数据访问需要特殊授权。
- **角色映射规则**：
  - 仓库管理员 -> 录入员。
  - 资产管理员/业务主管 -> 管理员。
  - 客服人员 -> 访客。
- **用户认证**：
  - MVP版本采用独立用户名密码认证方式。
  - 预留SSO集成扩展点，便于后续集成企业现有单点登录系统。
- **操作日志**：
  - 记录用户登录、票据录入/修改/作废、小箱状态变更、管理员配置修改等关键操作。
  - 每条日志包含操作人、操作时间、IP地址、操作内容、所属部门。
  - 日志需要保留至少1年，满足审计追溯要求。
  - 仅管理员可查询操作日志。
- **数据导出管控**：严格限制含敏感信息数据的导出权限，仅管理员和业务主管可导出，导出时自动脱敏。
- **数据库变更管理**：采用 migrations 版本管理工具（如Flyway、Liquibase）管理数据库变更，确保每个版本的数据库结构可追溯，支持正向升级和回滚，保障部署过程安全可控。

### 3.7 数据迁移模块
- **历史数据导入**：提供专门的导入工具，支持从旧系统批量迁移历史票据和小箱数据。
- **支持格式**：Excel(.xlsx)和CSV格式。
- **预处理校验**：导入前进行完整校验，包括必填项、格式、唯一性、部门合法性校验。
- **错误报告**：生成详细的迁移报告，包含成功条数、失败条数、错误行号及具体原因。
- **回滚支持**：支持迁移失败时整批次回滚，保证数据库一致性。
- **数据完整性**：迁移后确保票据、小箱、工序数据关联完整。

### 3.8 数据安全与合规
- **敏感数据定义**：票据中的客户名称、地址等个人信息定义为PII。
- **传输加密**：全站启用HTTPS。
- **存储安全**：对PII字段进行加密存储。
- **展示脱敏**：列表及详情展示进行掩码处理。
- **数据归档与保留策略**：
  - 实施"在线保存2年，历史归档"策略。
  - 对于超过2年的票据及工序数据，系统应提供归档机制（迁移至历史表或冷存储），以平衡数据库性能与存储成本，满足合规审计要求。
  - **[更新] 归档执行方式**：系统每月自动检查超过2年的数据，提示管理员确认后执行归档操作，兼顾自动化和可控性。
  - **[新增] 归档后查询**：归档后的历史数据仍需提供查询入口支持在线查询，满足审计和追溯需求，仅查询性能要求可适当放宽。
  - **[新增] 归档存储成本优化**：采用数据库分层存储方案，归档数据存储在低成本的归档存储介质（如AWS S3 Glacier、阿里云归档存储），平衡查询性能和存储成本，满足合规要求。
- **[新增] 归档数据恢复**：MVP版本不支持自动化的归档数据恢复操作。若确需恢复，由DBA手动执行数据库操作完成。后续迭代可根据业务需求再开发自动化恢复功能。
- **[新增] 数据备份与恢复**：
  - 采用托管数据库服务（如AWS RDS或云数据库）的自动备份功能。
  - 利用云平台提供的每日自动备份和时间点恢复能力，确保数据安全与业务连续性，无需自研备份模块。
- **操作日志保留**：操作日志需保留至少1年，满足审计要求。

### 3.9 运维监控模块
- **基础监控**：监控系统CPU/内存/磁盘使用率、数据库连接数/查询性能、关键API错误率。
- **告警机制**：针对异常情况（如系统错误率超过阈值、磁盘空间不足）设置告警，自动通知运维人员。
- **监控数据保留**：监控数据保留足够周期，支持趋势分析和问题排查。

18. **[新增] 离线录入功能**：MVP版本不需要支持离线录入，假设系统部署在企业内网，网络连接稳定可靠，所有操作均在线完成。
19. **[新增] 定期盘点功能**：MVP版本不需要提供专门的小箱盘点功能，小箱状态变更由仓库操作员手动维护，满足基本追溯需求即可。

21. **[新增] 完整修改痕迹审计**：MVP版本不需要记录每次修改的具体内容差异，仅记录修正时间和修正者即可，完整内容审计留待后续迭代。
22. **[新增] 数据字典在线配置**：MVP版本核心业务状态采用硬编码实现即可，不需要在线配置功能。

## 5. 设计考虑

- **一致性**：UI设计需严格遵循提供的CCS画面定义书。
- **语言规范**：UI使用中文；数据库/代码保留日文/英文。前端用户界面仅需中文，无需支持多语言切换。
- **易用性**：录入界面支持Tab键切换；状态流转自动化，减少人工干预。所有列表页面支持分页，避免一次性加载大量数据。关键字段支持模糊搜索，提升查询效率。
- **硬件适配**：优化条码扫描枪输入体验。
- **反馈性**：所有操作均需给出明确视觉反馈。
- **容错设计**：终端识别失败时提供降级方案；并发冲突提示明确。
- **性能体验**：合理分页与加载，结合归档策略保障长期性能。默认每页显示20条记录，用户可调整（20/50/100）。
- **权限设计**：遵循最小权限原则，实现功能权限与数据权限双层控制。
- **浏览器兼容性**：优先支持主流现代浏览器（Chrome、Edge、Firefox），要求支持Chrome 90+ 和 Edge 90+，不强制支持IE浏览器。
- **发布部署**：支持发布回滚机制，MVP版本至少支持全量发布后快速回滚到上一个稳定版本，降低新版本发布风险。

## 6. 技术考虑

- **部署架构**：优先部署在公有云（如AWS/Azure/阿里云），利用云服务商提供的数据库备份、监控等能力，降低运维成本。
- **前端框架**：React 或 Vue 主流版本。
- **后端架构**：Java (Spring Boot) 或 Node.js。
- **数据库**：
  - 关系型数据库。
  - 核心业务表增加 `version_no` 字段支持乐观锁。
  - 实施数据归档方案：设计历史数据表或对接冷存储服务，定期迁移2年以前的数据。
  - **[新增]** 启用托管数据库服务的自动备份与时间点恢复功能。
  - 关键字段添加索引优化查询性能。
  - **[新增]** 归档数据仍需支持查询，设计历史数据表结构保留查询能力。
  - **[新增]** 采用分层存储架构，归档数据迁移至低成本存储介质。
  - **[新增]** 采用 migrations 版本管理工具管理数据库变更，支持升级和回滚。
- **架构定位**：单一企业内部使用，按部门进行数据权限隔离即可，**不需要**多租户架构设计，降低设计和实现复杂度。
- **文件存储**：对象存储（AWS S3 或 MinIO）。
- **终端识别**：后端结合基础数据模块匹配。
- **安全性实现**：SSL证书、字段级加密、API鉴权、数据脱敏。
- **并发规模**：按支持50-100并发用户在线设计，数据库数据量规划为百万级票据记录。
- **性能优化**：通过索引优化、分页查询和归档策略满足性能要求。
- **数据权限实现**：在所有查询和写入操作中增加部门过滤条件，实现数据层面隔离。
- **浏览器兼容性**：针对主流现代浏览器（Chrome 90+, Edge 90+, Firefox）进行开发和测试，不强制支持IE浏览器。
- **草稿暂存**：MVP版本不实现会话超时自动草稿暂存功能，后续迭代按需开发。
- **监控告警**：需要基础系统监控和异常告警机制，保障系统高可用性。
- **发布回滚**：需要支持快速回滚机制，利用云平台部署能力（如容器滚动更新、蓝绿部署）实现快速回滚到上一个稳定版本。
- **数据字典实现**：MVP版本核心业务状态采用硬编码实现，简单可靠，满足当前需求。
