## 初始PRD

# 待办事项应用 MVP 产品需求文档 (PRD)

## 1. 概述

本项目旨在开发一款极简、高效的待办事项移动端应用。该产品主要面向个人用户，解决日常生活中任务遗忘、时间管理混乱的问题。通过提供核心的任务创建、管理和提醒功能，帮助用户专注于重要事项，提升个人生产力。

## 2. 项目目标

### 2.1 商业目标
*   在产品上线后3个月内，获得首批1,000名活跃用户。
*   保持应用的轻量化，降低维护成本，作为团队进入生产力工具领域的切入点。

### 2.2 用户目标
*   用户能够在10秒内快速记录一条待办事项。
*   用户不再遗漏重要的事项截止日期。
*   用户能够直观地查看今日需完成的所有任务。

## 3. 用户故事

**核心用户画像**：追求高效生活的职场新人或学生。

### 3.1 账户与设置
*   **故事 1**: As a *新用户*, I want to *使用邮箱或手机号快速注册账号*, so that 我的数据可以在不同设备间同步，避免数据丢失。
    *   **验收标准**:
        *   支持邮箱/密码注册。
        *   支持手机号/验证码注册。
        *   注册成功后自动登录并进入主界面。

### 3.2 任务管理
*   **故事 2**: As a *用户*, I want to *创建新的待办事项（包含标题、截止日期、优先级）*, so that 我能记录下需要做的事情并设定执行计划。
    *   **验收标准**:
        *   主界面有明显的“+”按钮。
        *   输入标题后点击“保存”即可创建任务（必填项仅为标题）。
        *   可选设置截止日期（默认为“今天”）。
        *   可选设置优先级（高/中/低，默认为“中”）。

*   **故事 3**: As a *用户*, I want to *查看我的待办列表（按时间或优先级排序）*, so that 我能了解当前有哪些任务需要处理。
    *   **验收标准**:
        *   默认展示所有未完成的任务。
        *   提供“今日”、“即将到来”、“已完成”三个筛选标签。
        *   高优先级任务在列表中通过颜色（如红色）醒目显示。

*   **故事 4**: As a *用户*, I want to *标记任务为已完成*, so that 我能追踪我的进度并获得成就感。
    *   **验收标准**:
        *   每个任务左侧有圆形复选框。
        *   点击复选框，任务划线并移入“已完成”列表。
        *   支持撤销完成操作。

*   **故事 5**: As a *用户*, I want to *编辑或删除已有的任务*, so that 我能应对计划变更或修正错误。
    *   **验收标准**:
        *   点击任务进入编辑页面，可修改所有字段。
        *   编辑页面底部有“删除”按钮。
        *   删除操作需二次确认，防止误触。

### 3.3 提醒功能
*   **故事 6**: As a *用户*, I want to *收到任务截止时间的通知提醒*, so that 我不会忘记重要事项。
    *   **验收标准**:
        *   在截止时间到达前15分钟（可配置）发送系统推送通知。
        *   用户需在系统设置中授权开启通知权限。

## 4. 功能需求

| 模块 | 功能点 | 优先级 | 描述 |
| :--- | :--- | :--- | :--- |
| **账户体系** | 注册/登录 | P0 | 支持手机号验证码登录（降低门槛），支持忘记密码找回。 |
| **任务管理** | 创建任务 | P0 | 必填：标题；选填：备注、日期、优先级。 |
| | 列表展示 | P0 | 支持按“今日/未来/已完成”分组展示。 |
| | 完成任务 | P0 | 点击复选框即完成，数据移入已完成列表。 |
| | 删除任务 | P0 | 支持删除，需二次确认。 |
| **提醒** | 系统通知 | P1 | 基于截止时间的本地推送通知。 |
| **设置** | 个人中心 | P2 | 修改昵称、头像、退出登录。 |

## 5. 非目标

以下功能**不在**本次MVP（最小可行性产品）范围内：
*   **协作功能**：不支持任务指派给他人或共享清单。
*   **子任务**：不支持在任务下拆分多层子任务。
*   **日历视图**：不支持以日历格式展示任务。
*   **语音输入**：不支持语音转文字创建任务。
*   **Web端/桌面端**：本次仅开发移动端。

## 6. 设计考虑

*   **极简主义**：界面保持干净，减少干扰元素，核心操作（添加任务）一键触达。
*   **手势交互**：考虑支持左滑删除、右滑完成的快捷手势（作为P1体验优化）。
*   **空状态设计**：当列表为空时，展示鼓励性插画与文案（如“太棒了，没有待办事项！”）。

## 7. 技术考虑

*   **技术栈**：
    *   前端：Flutter（实现iOS/Android跨平台，节省开发成本）。
    *   后端：Node.js + Express (或 Python FastAPI)。
    *   数据库：PostgreSQL (关系型数据库，适合结构化任务数据)。
*   **数据同步**：采用“本地优先”策略，本地SQLite先行响应，后台异步同步至云端，确保离线可用性。
*   **安全性**：用户密码需加盐哈希存储，接口采用JWT认证。
*   **约束**：后端服务需部署在云服务器（如AWS/Aliyun），预算控制在最低配额。

## 8. 成功指标

*   **次日留存率** > 30%。
*   **任务创建率**：平均每用户每周创建 > 5 个任务。
*   **任务完成率**：创建的任务中，标记为完成的比例 > 60%。

## 9. 开放问题与决策记录

*以下问题在生成过程中已根据行业最佳实践做出假设，开发前需确认：*

1.  **登录方式**：考虑到MVP开发速度，优先采用**手机号+验证码**登录（无需记忆密码），同时也支持邮箱注册作为备选。
2.  **排序逻辑**：默认排序规则确定为 **“优先级高置顶 > 截止时间近置顶”**。
3.  **离线模式**：确定为**支持离线查看与操作**，联网后自动同步，这是移动端Todo应用的基础体验要求。

## 迭代 1 - 分析问答

---
问题：任务的截止日期是强制的还是可选的？（PRD中提到“可选设置截止日期（默认为‘今天’）”，这存在逻辑矛盾：如果是可选的，就不应该有默认值强制填充；如果有默认值，则意味着所有任务都有日期。）
回答：截止日期应为可选。MVP应支持“收件箱”概念，允许用户快速记录想法而不必强制设定日期。默认值应为“无日期”，用户可主动选择设定为“今天”。这样更符合“10秒快速记录”的目标。
---
---
问题：应用启动时的默认视图是什么？（PRD目标中提到“直观查看今日任务”，但验收标准中写“默认展示所有未完成的任务”，两者存在冲突。）
回答：默认视图应为“今日”视图。这符合大多数用户每日规划的习惯，能直接呈现当天最相关的信息，减少认知负荷，与“直观查看今日需完成的所有任务”的目标保持一致。
---
---
问题：已完成任务的保留策略是什么？（PRD未说明已完成任务是永久保留还是定期清理，这会影响存储空间管理和列表性能。）
回答：MVP阶段采用“永久保留，手动清理”策略。已完成列表支持分页加载以保障性能。自动清理策略（如自动删除30天前的已完成任务）可作为P2功能在后续迭代中考虑。
---
---
问题：应用是否需要支持深色模式？
回答：需要。深色模式是现代移动端应用的基础体验，且Flutter框架支持成本低，应作为P1需求纳入，跟随系统设置自动切换。

    *   **验收标准**:
        *   支持邮箱/密码注册。
        *   支持手机号/验证码注册。
        *   注册成功后自动登录并进入主界面。

*   **故事 7**: As a *用户*, I want to *应用支持深色模式*, so that 我在光线较暗的环境下也能舒适使用，保护视力。
    *   **验收标准**:
        *   应用外观跟随系统设置自动切换深色/浅色模式。
        *   切换过程流畅，无UI错位或文字不可见问题。

*   **故事 8**: As a *用户*, I want to *注销我的账号并彻底删除个人数据*, so that 我能行使隐私权利，保障个人信息安全。
    *   **验收标准**:
        *   设置页面提供明显的“注销账号”入口。
        *   执行注销前需进行二次确认（如输入“确认注销”或输入密码）。
        *   后端在收到请求后，需在规定时间内彻底清除用户的所有个人数据。

### 3.2 任务管理
*   **故事 2**: As a *用户*, I want to *创建新的待办事项（包含标题、截止日期、优先级）*, so that 我能记录下需要做的事情并设定执行计划。
    *   **验收标准**:
        *   主界面有明显的“+”按钮。
        *   输入标题后点击“保存”即可创建任务（必填项仅为标题）。
        *   截止日期默认为“无日期”（支持“收件箱”概念，允许快速记录想法）。
        *   可选设置截止日期（用户可主动选择设定为“今天”）。
        *   可选设置优先级（高/中/低，默认为“中”）。

*   **故事 3**: As a *用户*, I want to *查看我的待办列表（按时间或优先级排序）*, so that 我能了解当前有哪些任务需要处理。
    *   **验收标准**:
        *   应用启动时默认展示“今日”视图。
        *   提供“今日”、“收件箱”（无日期）、“即将到来”、“已完成”筛选标签。
        *   高优先级任务在列表中通过颜色（如红色）醒目显示。

*   **故事 4**: As a *用户*, I want to *标记任务为已完成*, so that 我能追踪我的进度并获得成就感。
    *   **验收标准**:
        *   每个任务左侧有圆形复选框。
        *   点击复选框，任务划线并移入“已完成”列表。
        *   支持撤销完成操作。
        *   已完成任务永久保留，直至用户手动清理。

*   **故事 5**: As a *用户*, I want to *编辑或删除已有的任务*, so that 我能应对计划变更或修正错误。
    *   **验收标准**:
        *   点击任务进入编辑页面，可修改所有字段。
        *   编辑页面底部有“删除”按钮。
        *   删除操作需二次确认，防止误触。

### 3.3 提醒功能
*   **故事 6**: As a *用户*, I want to *收到任务截止时间的通知提醒*, so that 我不会忘记重要事项。
    *   **验收标准**:
        *   在截止时间到达前15分钟（可配置）发送系统推送通知。
        *   用户需在系统设置中授权开启通知权限。

## 5. 非目标

以下功能**不在**本次MVP（最小可行性产品）范围内：
*   **协作功能**：不支持任务指派给他人或共享清单。
*   **子任务**：不支持在任务下拆分多层子任务。
*   **重复任务**：不支持每日打卡、每周例会等循环任务逻辑。
*   **日历视图**：不支持以日历格式展示任务。
*   **语音输入**：不支持语音转文字创建任务。
*   **法律条款页面**：应用内不包含独立的隐私政策和服务条款页面。
*   **Web端/桌面端**：本次仅开发移动端。
*   **自动清理**：不支持自动删除过期的已完成任务（需手动清理）。

## 6. 设计考虑

*   **极简主义**：界面保持干净，减少干扰元素，核心操作（添加任务）一键触达。
*   **手势交互**：考虑支持左滑删除、右滑完成的快捷手势（作为P1体验优化）。
*   **空状态设计**：当列表为空时，展示鼓励性插画与文案（如“太棒了，没有待办事项！”）。
*   **深色模式**：作为现代移动应用基础体验，需设计完整的深色模式UI规范，自动适配系统设置。

## 7. 技术考虑

*   **技术栈**：
    *   前端：Flutter（实现iOS/Android跨平台，节省开发成本，原生支持深色模式）。
    *   后端：Node.js + Express (或 Python FastAPI)。
    *   数据库：PostgreSQL (关系型数据库，适合结构化任务数据)。
*   **数据统计**：集成 Firebase Analytics 或友盟 SDK，在注册完成、任务创建、任务完成、应用启动等关键节点进行埋点，用于量化产品成功指标。
*   **数据同步**：采用“本地优先”策略，本地SQLite先行响应，后台异步同步至云端，确保离线可用性。
*   **性能优化**：已完成任务列表需实现分页加载机制，防止长期使用后数据量过大导致列表卡顿。
*   **安全性与合规**：
    *   用户密码需加盐哈希存储，接口采用JWT认证。
    *   后端需实现账号注销接口，确保能彻底清除用户个人数据，符合GDPR及个人信息保护法要求。
*   **约束**：后端服务需部署在云服务器（如AWS/Aliyun），预算控制在最低配额。

## 8. 成功指标

*   **次日留存率** > 30%。
*   **任务创建率**：平均每用户每周创建 > 5 个任务。
*   **任务完成率**：创建的任务中，标记为完成的比例 > 60%。
*   **数据统计方式**：通过集成第三方数据统计SDK（如Firebase Analytics或友盟）进行自动埋点统计，确保数据准确性。

## 9. 开放问题与决策记录

*以下问题已根据最新讨论确认：*

1.  **登录方式**：考虑到MVP开发速度，优先采用**手机号+验证码**登录（无需记忆密码），同时也支持邮箱注册作为备选。
2.  **排序逻辑**：默认排序规则确定为 **“优先级高置顶 > 截止时间近置顶”**。
3.  **离线模式**：确定为**支持离线查看与操作**，联网后自动同步，这是移动端Todo应用的基础体验要求。
4.  **截止日期逻辑**：确定为**可选，默认为“无日期”**。支持“收件箱”概念，允许用户快速记录想法而不必强制设定日期，符合“10秒快速记录”目标。
5.  **默认视图**：确定为**“今日”视图**。符合大多数用户每日规划习惯，直观呈现当天最相关信息。
6.  **已完成任务保留策略**：MVP阶段采用**“永久保留，手动清理”**策略。已完成列表支持分页加载以保障性能。自动清理策略（如自动删除30天前的已完成任务）作为P2功能规划。
7.  **深色模式**：确定为**支持**。深色模式是现代移动端应用基础体验，Flutter框架支持成本低，作为P1需求纳入，跟随系统设置自动切换。
8.  **重复任务**：MVP版本**不支持**。重复任务涉及复杂的逻辑判断，不属于核心范围，后续版本再考虑迭代。
9.  **法律条款页面**：MVP版本**不需要**包含独立的隐私政策和服务条款页面。
10. **账号注销**：确定为**必须支持**。依据数据隐私合规要求，应用必须提供用户主动注销账号的入口，并实现数据彻底清除。
11. **数据埋点**：需集成第三方统计SDK以量化成功指标，在关键用户行为节点进行埋点。
