# PRD 转换为 Ralph JSON 格式

## 你的任务

把已完成的 PRD 转换为 `prd.json` 格式，给 Ralph 自动化执行系统使用。

---

## 输出格式要求：

```json
{
  "project": "[项目名称]",
  "branchName": "ralph/[功能名称-短横线连接]",
  "description": "[项目描述]",
  "userStories": [
    {
      "id": "US-001",
      "title": "[故事标题]",
      "description": "作为 [用户], 我想要 [功能], 以便 [收益]",
      "acceptanceCriteria": [
        "验收标准 1",
        "验收标准 2",
        "类型检查通过"
      ],
      "priority": 1,
      "passes": false,
      "notes": ""
    }
  ]
}
```

---

## 故事大小：第一重要规则

**每个故事必须能在**一次 **Ralph 迭代（一个上下文窗口）中完成。**

Ralph 每次迭代都是一个全新的 Amp 实例，没有之前工作的记忆。如果故事太大，大语言模型上下文会溢出，生成破碎代码。

### 大小合适的故事：
- 给数据表添加一列并创建迁移文件
- 在已有页面添加一个 UI 组件
- 更新一个服务端操作添加新逻辑
- 给列表添加筛选下拉框

### 太大了，必须拆分：
- "构建整个仪表盘" → 拆分成：数据结构、查询、UI 组件、筛选功能
- "添加用户认证" → 拆分成：数据结构、中间件、登录界面、会话处理
- "重构 API" → 每个接口或模式拆分成一个故事

**经验法则：** 如果你不能用 2-3 句话描述这个改动，它就太大了。

---

## 故事排序：依赖在前

故事按优先级顺序执行。前面的故事不能依赖后面的故事。

**正确顺序：**
1. 数据结构/数据库 改动（迁移文件）
2. 服务端操作 / 后端 逻辑
3. 使用后端的 UI 组件
4. 聚合数据的仪表盘/汇总视图

**错误顺序：**
1. UI 组件（依赖还不存在的数据表）
2. 数据结构 修改

---

## 验收标准：必须可验证

每条标准都必须是 Ralph 可以检查通过的，不能含糊。

### 好的标准（可验证）：
- "给任务表添加 `status` 字段，默认值为 'pending'"
- "筛选下拉框选项：全部、进行中、已完成"
- "点击删除按钮显示确认对话框"
- "类型检查通过"
- "测试通过"

### 坏的标准（模糊）：
- "工作正常"
- "用户可以轻松操作"
- "用户体验好"
- "处理边界情况"

### 总是添加：
```
"类型检查通过"
```

对于有可测试逻辑的故事，还要添加：
```
"测试通过"
```

对于修改 UI 的故事，还要添加：
```
"使用 dev-browser 技能在浏览器中验证"
```

前端故事在视觉验证通过之前不算完成。Ralph 会使用 dev-browser 技能导航到页面，交互 UI，确认改动工作正常。

---

## 转换规则

1. **每个用户故事对应一个 JSON 条目**
2. **ID**：顺序编号（US-001, US-002, ...）
3. **优先级**：基于依赖顺序，文档顺序，数字越小优先级越高
4. **所有故事**：设置 `passes: false` 和 `notes: ""`
5. **branchName**：从功能名派生，使用短横线连接(kebab-case)，前缀 `ralph/`
6. **总是添加**：`"类型检查通过"` 到每个故事的验收标准

---

## 拆分大 PRD

如果 PRD 有大功能，拆分它：

**原文：**
> "添加用户通知系统"

**拆分为：**
1. US-001: 在数据库添加通知数据表
2. US-002: 创建发送通知的通知服务
3. US-003: 在页面头部添加通知铃铛图标
4. US-004: 创建通知下拉面板
5. US-005: 添加标记已读功能
6. US-006: 添加通知偏好设置页面

每个都是一个聚焦的改动，可以独立完成并验证。
