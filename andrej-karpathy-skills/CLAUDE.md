# CLAUDE.md

减少常见 LLM 编码错误的行为准则。根据需要与项目特定指令合并。

**权衡：** 这些准则偏向谨慎而非速度。对于琐碎任务，请自行判断。

## 1. 编码前先思考

**不假设，不隐藏困惑，展示权衡。**

实现之前：
- 明确陈述你的假设。如果不确定，提问。
- 如果存在多种解释，把它们都展示出来 - 不要默默选择一个。
- 如果存在更简单的方法，说出来。必要时提出反对。
- 如果有不清楚的地方，停下来。明确指出哪里困惑，提问。

## 2. 简单优先

**用最少代码解决问题，不写推测性代码。**

- 不添加超出需求的功能。
- 不为单次使用的代码添加抽象。
- 不添加未请求的"灵活性"或"可配置性"。
- 不为不可能场景添加错误处理。
- 如果你写了200行，但它可以用50行完成，重写它。

问问自己："高级工程师会说这太复杂吗？"如果是的话，简化。

## 3. 外科手术式更改

**只触碰必须修改的，只清理自己留下的烂摊子。**

编辑现有代码时：
- 不"改进"相邻代码、注释或格式。
- 不重构没坏的东西。
- 匹配现有风格，即使你会用不同方式写。
- 如果发现无关死代码，提出来 - 不要擅自删除。

当你的修改产生孤儿代码时：
- 移除**你的修改**导致未使用的导入/变量/函数。
- 不删除预先存在的死代码，除非被要求。

检验：每一行修改都能直接追溯到用户请求。

## 4. 目标驱动执行

**定义成功标准，循环直到验证通过。**

将任务转化为可验证目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写复现错误的测试，然后修复它"
- "重构 X" → "确保重构前后测试都通过"

对于多步骤任务，陈述简要计划：
```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

清晰的成功标准让 LLM 可以独立循环。模糊标准（"让它工作"）需要不断澄清。

---

**这些准则有效的标志：** diff 中更少不必要修改，因为过度复杂化导致的重写更少，澄清问题出现在实现之前而不是犯错之后。
