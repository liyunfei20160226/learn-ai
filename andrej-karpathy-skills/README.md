# Karpathy 启发的 Claude Code 行为准则

> **正在寻找托管智能体平台？** 看看 [Multica](https://github.com/multica-ai/multica) — 一个运行和管理可复用技能编码智能体的开源平台。

基于 [Andrej Karpathy 的观察](https://x.com/karpathy/status/2015883857489522876) 对 LLM 编码缺陷的总结，改善 Claude Code 行为的单个 `CLAUDE.md` 文件。

## 存在的问题

来自 Andrej 的帖子：

> "模型会替你做出错误假设，并且一直运行下去而不检查。它们不管理自己的困惑，不寻求澄清，不暴露不一致性，不展示权衡取舍，在该反对的时候不反对。"

> "它们真的很喜欢过度复杂化代码和 API，过度膨胀抽象，不清理死代码......用1000行实现本可以用100行解决的问题。"

> "它们有时仍然会作为副作用，修改/删除它们并不完全理解的注释和代码，即使这些修改与任务正交。"

## 解决方案

四个原则直接解决这些问题：

| 原则 | 解决问题 |
|-----------|-----------|
| **编码前先思考** | 错误假设、隐藏困惑、缺少权衡 |
| **简单优先** | 过度复杂、膨胀的抽象 |
| **外科手术式更改** | 正交编辑、触碰不该碰的代码 |
| **目标驱动执行** | 通过测试优先，可验证的成功标准 |

## 四大原则详解

### 1. 编码前先思考

**不假设，不隐藏困惑，展示权衡。**

LLM 经常默默地选择一个解释就继续了。这个原则迫使显式推理：

- **明确陈述假设** — 如果不确定，提问而不是猜测
- **展示多种解释** — 存在歧义时不要默默选择一个
- **必要时提出反对** — 如果有更简单的方法，说出来
- **困惑时停下来** — 明确指出什么不清楚，请求澄清

### 2. 简单优先

**用最少代码解决问题，不写推测性代码。**

对抗过度工程化倾向：

- 不添加超出需求的功能
- 不为单次使用的代码添加抽象
- 不添加未请求的"灵活性"或"可配置性"
- 不为不可能场景添加错误处理
- 如果200行可以简化为50行，重写它

**检验：** 高级工程师会说这太复杂吗？如果是的话，简化。

### 3. 外科手术式更改

**只触碰必须修改的，只清理自己留下的烂摊子。**

编辑现有代码时：

- 不"改进"相邻代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格，即使你会用不同方式写
- 如果发现无关死代码，提出来 — 不要擅自删除

当你的修改产生孤儿代码时：

- 移除**你的修改**导致未使用的导入/变量/函数
- 不删除预先存在的死代码，除非被要求

**检验：** 每一行修改都能直接追溯到用户请求。

### 4. 目标驱动执行

**定义成功标准，循环直到验证通过。**

将命令式任务转化为可验证目标：

| 不要这样 | 要转化为 |
|--------------|-----------------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写复现错误的测试，然后修复它" |
| "重构 X" | "确保重构前后测试都通过" |

对于多步骤任务，陈述简要计划：

```
1. [步骤] → 验证：[检查点]
2. [步骤] → 验证：[检查点]
3. [步骤] → 验证：[检查点]
```

清晰的成功标准让 LLM 可以独立循环。模糊标准（"让它工作"）需要不断澄清。

## 安装

**选项 A: Claude Code 插件（推荐）**

在 Claude Code 中，先添加市场：
```
/plugin marketplace add forrestchang/andrej-karpathy-skills
```

然后安装插件：
```
/plugin install andrej-karpathy-skills@karpathy-skills
```

这会将准则安装为 Claude Code 插件，使技能在所有项目中可用。

**选项 B: CLAUDE.md（每个项目）**

新项目：
```bash
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

现有项目（追加）：
```bash
echo "" >> CLAUDE.md
curl https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md >> CLAUDE.md
```

## 核心洞见

来自 Andrej：

> "LLM 在循环直到达到特定目标方面异常出色......不要告诉它做什么，给它成功标准，看它自己前进。"

"目标驱动执行"原则抓住了这一点：将命令式指令转化为带有验证循环的声明式目标。

## 如何判断是否生效

这些准则有效时，你会看到：

- **diff 中更少不必要修改** — 只出现请求的修改
- **因为过度复杂化导致的重写更少** — 第一次代码就简洁
- **实现前就有澄清问题** — 不是犯错之后才问
- **干净、最小化的 PR** — 没有顺路重构或"改进"

## 自定义

这些准则设计为可以与项目特定指令合并。将它们添加到你现有的 `CLAUDE.md` 或创建一个新的。

对于项目特定规则，添加这样的章节：

```markdown
## 项目特定指南

- 使用 TypeScript strict 模式
- 所有 API 端点必须有测试
- 遵循 `src/utils/errors.ts` 中的现有错误处理模式
```

## 权衡说明

这些准则偏向**谨慎而非速度**。对于琐碎任务（简单拼写修复，明显的单行修改），请自行判断 — 不是每个修改都需要完整的严谨性。

目标是减少非平凡工作中代价高昂的错误，而不是减慢简单任务。
